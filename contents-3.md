1. 랄프 루프(Ralph Loop) - 2026년을 정의할 AI 코딩 기법

2026년이 시작된 지 일주일도 안 되어, 테크 트위터는 “랄프 위검 루프(Ralph Wiggum Loop)“에 대한 논의로 뜨겁게 달아올랐다. 심슨 가족의 “I’m in danger” 밈으로 유명한 그 사랑스럽게 끈질긴 캐릭터의 이름을 딴 이 기법은, 우리가 자율 AI 개발에 대해 생각하는 방식에 도전장을 내밀고 있다.

Geoffrey Huntley가 처음 만든 이 기법은 놀랍도록 단순하면서도 강력하다:

while :; do cat PROMPT.md | agent ; done
 코드 복사
이게 전부다. AI 코딩 에이전트에 같은 프롬프트를 반복적으로 넣는 무한 루프.

핵심은 이것이다: 진행 상황은 LLM의 컨텍스트 윈도우에 저장되지 않는다. 대신 파일과 git 히스토리에 저장된다. 컨텍스트가 꽉 차면, 새로운 에이전트가 새 컨텍스트로 시작해서 이전 에이전트가 멈춘 곳에서 이어받는다.

Huntley의 표현: “그게 랄프의 아름다움이다 - 이 기법은 비결정론적 세계에서 결정론적으로 나쁘다.”


2. OpenAI 하니스 엔지니어링 기고글

1️⃣ 컨텍스트 엔지니어링 — AI한테 뭘 알려줄 것인가 신입사원한테 아무 설명 없이 "일해"라고 하면 못 합니다. AI도 똑같습니다. 프로젝트 규칙, 코딩 컨벤션, 작업 현황... 이런 정보를 CLAUDE.md, AGENTS.md 같은 문서로 정리해서 넘겨줘야 합니다. 에이전트가 읽을 수 없는 정보는 존재하지 않는 것과 같으니까요.

2️⃣ 아키텍처 제약 — AI가 하면 안 되는 것을 정하기 볼링장 범퍼처럼, 자유를 주면 삽질하고, 경계를 정해주면 오히려 더 빨리 정답에 도달합니다. 역설적이지만, 제약이 생산성을 높입니다.

3️⃣ 가비지 컬렉션 — AI가 만든 쓰레기 치우기 AI는 좋은 패턴도 복제하지만, 나쁜 패턴도 복제합니다. 정리하지 않으면 품질은 시간이 갈수록 떨어집니다. OpenAI 팀은 처음에 매주 금요일을 "AI 슬롭 청소일"로 썼다가, 결국 자동 정리 시스템을 하네스에 내장했습니다.
